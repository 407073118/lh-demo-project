## 竞品定位反推

### 聚宽：策略 IDE 优先

聚宽的核心界面心智是“在线策略编辑器”。公开新手指引里的流程非常清楚：策略列表进入策略详情，在线编辑 Python 策略，设置回测参数，先做快速编译/快速回测，再进入完整回测结果，最后把策略推进到模拟交易。

可反推出的 UI 结构：

- 顶部是很轻的全局导航，用户注意力主要放在编辑器和运行区。
- 主体是左右分栏：左侧深色代码编辑器，右侧参数、快速结果和日志。
- 运行入口不是单一按钮，而是“保存 / 编译运行 / 运行完整回测”的渐进验证链。
- 完整结果页有左侧结果目录，顶部是核心指标，主区是收益、基准、成交、回撤等组合图。
- 聚宽的设计重点不是漂亮，而是降低从代码到回测结果的摩擦。

对 LH Quant 的启发：编辑态必须像 IDE，而不是像表单页。回测前应该强调代码、参数、校验、日志；回测后才进入结果分析。

### 米筐：研究平台优先

米筐的核心心智更像“研究平台 + 回测报告 + 模拟交易闭环”。公开文档展示了代码策略、向导式策略生成器、回测结果、模拟交易、因子研究和结果数据 API。

可反推出的 UI 结构：

- 代码策略页也是左右分栏：左侧代码，右侧参数/自定义参数/回测摘要。
- 自定义参数能从代码声明映射到界面表格，说明参数不是散落表单，而是策略配置的一部分。
- 回测结果页有明确进度条和状态，右侧提供“编辑策略”“下载日志”等动作。
- 结果页采用横向 tab：收益概览、交易详情、持仓详情、账户信息、输出日志、性能分析。
- 收益概览不是孤立收益图，而是同时展示回测收益、基准收益、超额收益、每日盈亏、买卖点等上下文。
- 向导式策略生成器使用“左侧因子/指标库，右侧我的指标/规则”的购物车式交互，适合低代码配置复杂策略。
- 模拟交易从完整分钟回测结果页启动，启动后保留运行状态、持仓、成交和资金使用信息。

对 LH Quant 的启发：结果页必须是报告，而不只是图表。数据、交易、持仓、日志、性能和血缘要有稳定位置。

## 共同 UX 模式

1. 研究闭环是第一原则。
   用户路径应是：选择/编辑策略 -> 配置数据和执行假设 -> 校验/快速运行 -> 完整回测 -> 分析结果 -> 复盘历史/进入模拟。

2. 参数不是普通表单。
   参数要绑定策略、版本、代码快照、执行假设和数据范围。用户改了参数后，结果应该明确标记为“已过期，需要重新运行”。

3. 回测结果需要可审计。
   只给累计收益和 K 线不够。专业用户会追问：基准是什么、复权方式是什么、撮合价是什么、是否限制成交量、缺失交易日怎么处理、手续费和滑点如何设置。

4. 图表要服务判断。
   主图需要可以读出策略收益、基准收益、超额收益、回撤、每日盈亏、买卖点和成交量。交易细节应该能和图表相互定位。

5. 日志和错误是主界面的一部分。
   聚宽和米筐都把运行错误、输出日志、性能分析放在工作流内。LH Quant 不能只在失败时弹错误，应持续展示校验、数据加载、缓存命中和落库过程。

6. 结果页要有推进动作。
   竞品结果页不是终点，后续动作包括编辑策略、下载日志、启动模拟、复用历史回测、查看明细。

## 当前 LH Quant 的结构性问题

当前实现已经比早期截图更接近工作台：有深色顶部导航、结果模式、运行上下文、图表和抽屉。烟测也覆盖了桌面和移动端截图。但仍然有几个“别扭”的根因。

### 1. 第一屏像 dashboard，不像投研终端

当前结果页顶部是 run context、5 个指标卡、一个“指标明细”折叠区，然后主图。指标卡占用强视觉权重，但用户最需要的“数据可信度、基准、执行假设、配置快照”被压到小字或抽屉里。

建议：第一屏改为“运行身份条 + 主图 + 可信度侧栏”。指标可以保留，但做成紧凑指标带或可展开指标矩阵，不要每个都像一个大卡片。

### 2. 结果页缺少报告型信息架构

当前 `ResultDashboard` 把概览、价格信号、收益回撤、运行明细串在同一长页面。它比隐藏在顶层 tab 后好，但还没有形成米筐式报告页。

建议：保留 dashboard-first，但下方用稳定 tab 或左侧目录组织：概览、收益风险、价格信号、持仓暴露、订单成交、数据血缘、日志、性能。

### 3. 编辑态和结果态的模式切换不够强

聚宽的编辑态很明确：代码编辑是主角，右侧是参数和运行结果。LH Quant 当前编辑态已经有 `ResearchWorkbench`，但像预览面板，不像真正工作区。右侧运行设置和下方输出的关系还不够紧。

建议：编辑态变成三段式：左资源树、中代码/策略说明、右配置面板、底部输出控制台。运行前不展示伪分析，只展示校验和日志。

### 4. 数据可信度没有变成视觉优先级

LH Quant 的方向是本地数据闭环和数据库事实源，但 UI 中“数据库已连接、数据源、复权、缓存”仍像状态装饰。量化用户真正关心的是：这次回测的数据是否完整，实际来源是什么，缺口在哪里，为什么 fallback。

建议：每次结果必须有一个“数据可信度面板”，显示 provider chain、实际 provider、复权、覆盖率、缺失日期、数据版本、引擎假设。

### 5. 风险语义需要更严谨

A 股红涨绿跌是对的，但最大回撤这种风险指标用绿色显示，会让“风险被美化”。收益可以用红，亏损/回撤可以用风险色或中性黑，不应简单套市场上涨/下跌语义。

建议：拆分两套色彩语义：

- 市场语义：红=上涨/买入，绿=下跌/卖出。
- 风险语义：风险、回撤、失败、缺失数据使用橙/红/中性强调，不沿用市场绿。

### 6. 移动端不是核心，但不能破坏信任

烟测移动端截图中 run context 横向内容被裁切感较强。量化工作台移动端不需要完整编辑体验，但至少应可看结果摘要、运行状态、关键图表和日志。

建议：移动端进入只读结果模式，参数和历史通过抽屉打开，run id 和时间换行显示。

## 反推后的界面蓝图

### 全局框架

```text
-------------------------------------------------------------+
| Brand | 数据 | 研究 | 回测 | 模拟交易 | 策略库 | 社区 | 状态 |
+-------------------------------------------------------------+
| 编辑态：资源树 | 代码/Notebook/输出 | 参数/校验/运行       |
| 结果态：                 结果报告主画布 + 可开关抽屉       |
+-------------------------------------------------------------+
```

顶部导航保留，但不要让它成为视觉主角。量化平台的主角是当前策略和当前 run。

### 编辑态

```text
左：资源树
  策略列表
  最近运行
  数据资产

中：研究工作区
  strategy.py / notebook / factor.py
  代码编辑或策略说明
  底部：控制台 / 运行日志 / 结果预览 / 产物

右：运行配置
  基础：标的、日期、频率、基准
  数据：provider、复权、缺失策略
  策略：参数、自定义参数、约束
  执行：撮合、滑点、手续费、成交量限制
  风险：仓位、止损、最大回撤停止
  主按钮：校验 / 快速回测 / 完整回测
```

核心交互：

- 参数变化后，当前结果出现“配置已改变”标记。
- 校验失败时定位到具体配置项，不只在面板底部给一段错误。
- 快速回测输出在右侧或底部，完整回测进入结果报告页。

### 结果态

```text
运行身份条
  策略 / 标的 / 区间 / 基准 / 频率 / 数据源 / 引擎 / run id / 状态

第一屏
  左：主分析图
      策略收益、基准收益、超额收益、回撤、每日盈亏、买卖点
  右：可信度与操作
      数据覆盖、实际来源、复权、撮合假设、日志、编辑策略、复制配置、启动模拟

下方报告
  概览 | 收益风险 | 价格信号 | 持仓暴露 | 订单成交 | 数据血缘 | 日志 | 性能
```

核心交互：

- 鼠标悬停图表时同步高亮当日交易、持仓、权益点。
- 点击买卖点打开对应订单/成交明细。
- 点击数据缺口进入数据血缘面板。
- 从历史回测打开后，配置、代码快照和图表应一起恢复。

## 组件反推清单

| 模块 | 米筐/聚宽模式 | LH Quant 应该落地 |
| --- | --- | --- |
| 顶部导航 | 暗色、轻量、功能入口 | 保留深色平台条，但当前 run 状态更突出 |
| 策略编辑 | 左代码右参数 | 编辑态强化代码/策略说明，中区不要像普通卡片 |
| 参数面板 | 基础 + 高阶 + 自定义参数 | 用分组 schema 渲染，不要把所有字段平铺 |
| 运行动作 | 保存、编译/快速、完整运行 | 增加校验和快速运行层级 |
| 结果摘要 | KPI 行 + 图表 | KPI 压缩，主图和可信度优先 |
| 结果明细 | 横向 tab 或左侧目录 | 稳定报告目录，避免长页面堆叠 |
| 数据可信度 | 数据、复权、撮合、风险数据可查 | 做成第一屏侧栏和专门血缘页 |
| 日志错误 | 运行错误、输出日志、性能分析 | 日志是常驻工作流，不是失败后才显示 |
| 模拟交易 | 从完整回测推进 | 结果页提供“启动模拟”占位动作 |
| 因子/向导 | 左指标库右我的指标 | 后续做因子策略时使用购物车式选择器 |

## 第一阶段改版建议

第一阶段不需要重做全部平台，先解决“别扭”的主因。

1. 重排结果页第一屏。
   把 `ResultOverview` 从大指标卡改成紧凑指标带；把“数据可信度 / 运行假设 / 后续操作”放到主图右侧；去掉空的“指标明细”占位。

2. 强化 run context。
   显示 benchmark、frequency、actual provider、adjust、engine version、data version、run id。长字段允许换行或复制，不要挤在一行。

3. 建立报告目录。
   在当前 dashboard-first 基础上加入结果报告导航：概览、收益风险、价格信号、持仓、订单成交、数据血缘、日志。

4. 编辑态补齐运行层级。
   运行按钮拆为“校验配置”“快速回测”“完整回测”。暂时可以只实现 UI 状态，不一定马上接完整异步后端。

5. 把风险语义从市场红绿里拆出来。
   最大回撤、数据缺失、运行失败、覆盖率不足统一使用风险色或中性色，避免绿色表达坏消息。

6. 更新 `apps/web/DESIGN.md`。
   现有设计契约方向对，但需要补充竞品反推后的信息架构、结果报告页结构和风险/市场色彩分离规则。

## 不建议照抄的部分

- 不建议复刻老式网页表格的视觉细节。聚宽和米筐的老截图功能结构强，但视觉年代感较重。
- 不建议第一版就做完整 Notebook、因子商城、模拟交易状态机。LH Quant 当前更需要先把单次回测做可信。
- 不建议把所有结果都藏进 tab。第一屏必须直接回答“这次结果怎么样、可信吗、下一步做什么”。
- 不建议做营销化首页。这个产品应直接进入工作台。
